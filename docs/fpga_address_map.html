SGI IP22 (Indy/Indigo2) Physical Address Map & FPGA Bring-up

Reference for the r9999 MIPS-III OOO core booting 64-bit MIPS Linux (and, later, IRIX) on an SGI IP22 personality — in the C++/Verilator simulator, in the standalone interp_mips functional sim, and on the Zynq UltraScale FPGA. On FPGA the CPU's physical requests are routed by hdl/axi_is_the_worst_v1_0_M00_AXI.v into DRAM shared with the ARM PS, and the ARM driver (naive-driver-for-rv64-on-ultra96v2/) stages images and emulates the slow devices. Sources: indy_docs/mc.pdf (MC Chip Spec §4, p.22 — authoritative), indy_docs/hpc3.pdf, sgi_indy.hh (compute_mem_range_type), Linux arch/mips/sgi-ip22, MAME src/mame/sgi/mc.cpp, and the IP32/IP24 PROM disassemblies. All addresses are physical unless noted.

Status (this branch): RTL boots IP22 Linux to NR_IRQS; standalone interp_mips boots to device init (MC reg 0x1c). FPGA bring-up = ARM driver + device emulation + a small RTL decode update. The address map below is the canonical contract shared by the RTL, the ARM driver's fpga_remap(), and the ARM device emulator.

Three facts from the spec drive the whole FPGA design:

  • Max 512 MB RAM, in two 256 MB banks — Low Local @ 0x08000000 and High System @ 0x20000000. Only the low 512 MB is reachable through kseg0/kseg1; the High bank needs the TLB.
  • The bottom 512 KB is an alias of the start of RAM (0x08000000–0x0807FFFF), provided "for the CPU exception vectors at 0x00000000 and 0x00000080" and for EISA/ISA DMA that lives in the first 640 KB. So PA 0x1000 and PA 0x08001000 are the same DRAM cell. This is why the kernel reserves the 0x4000 gap below _text.
  • "Accesses to unused or reserved regions cause a bus error on reads and a bus error interrupt on writes." Documented — but see §8: we do not need to implement a guest bus-error path.

3. The core problem: 4 GB sparse → <512 MB aperture (the compaction)

The IP22 owns a 4 GB physical space but the ARM↔softcore shared-DRAM window is <512 MB (the driver maps 496 MB). You cannot map the 4 GB 1:1. The address translation is therefore a fixed, sparse compaction MMU that folds the few regions the guest actually backs into the aperture and treats everything else as inert. Of the 4 GB only ~528 MB is ever real (512 MB RAM in two banks + 16 MB device window); the rest is bus-error by spec.

The compaction is one contract that must be byte-identical in three places: (1) M00_AXI.v (the softcore's own accesses), (2) fpga_remap() in the ARM driver's loadelf.cc (staging images), and (3) the ARM device/DMA emulator (when it dereferences a guest physical address — a SCSI descriptor, an ARC pointer). Drift here silently corrupts. fpga_remap() should be the canonical function, matched bit-for-bit by the Verilog.

RAM budget (the only real design variable): 496 MB comfortably holds one 256 MB bank + the 16 MB device window. Cap at the Low Local bank (≤256 MB) for bring-up — it is kseg0-reachable, so RAM needs no TLB datapath. The High bank stays unmapped (correctly faults a size probe) and only matters if you ever want >256 MB, which also drags in the TLB — a natural "later".

5. Boot architecture: preserve the 0xBFC00000 ROM (for IRIX)

IRIX boots through the real ARC PROM (arcsboot/sash → unix), so we keep a PROM at the reset vector rather than jumping the core straight to a kernel entry. Linux and IRIX then share one entry model.

  • SPB @ PA 0x1000 (must stay there) — small: magic + a pointer to the romvec (now up in the PROM).
  • PROM @ 0xBFC00000 (PA 0x1FC00000, DRAM-backed) — reset vector + romvec table + ARC stubs. Currently arcs_fw.S is one blob linked at 0xA0001000; split it: SPB at 0x1000, code at 0xBFC00000.
  • Reset vector job — now (Linux): minimal CP0 init, set up ARC argc/argv/envp + cmdline, jump to kernel_entry (0x88300D58). later (IRIX/full): the real ARC disk boot, or a faithful SGI PROM image (the ip32prom-decompiler proves a reassemblable PROM is feasible).

Boot ROM is read-mostly code in DRAM — no PL device needed. The driver keeps resume_pc=0xBFC00000.

6. FPGA translation: CPU physical → DRAM (M00_AXI & fpga_remap)

The ARM PS sets baseaddr (DRAM window base = the ioctl'd phys_addr), addrmask, sgi_mode=1, resume_pc, then releases reset. Each request becomes w_axi_addr = baseaddr + t_cpuaddr. The current sgi_mode translation:

The driver's fpga_remap() matches this bit-for-bit (verified). Two corrections from this thread:

  1. Implement the sys_mem_alias in both HW and fpga_remap, so PA 0–0x7FFFF folds onto the RAM bottom (one copy, like real HW): 
    if      (addr <= 0x0007ffff) addr += 0x08000000;   // alias → real RAM
else if (addr <= 0x17ffffff) addr &= 0x0fffffff;   // RAM
else if (0x1f000000..0x1fffffff) addr = (addr&0xffffff) + 0x10000000; // devices
    Then the SPB physically lives at DRAM+0x08001000 and the vectors at DRAM+0x08000000 (inside the kernel's reserved 0x4000 gap) — exactly like IP22, and the ARM load + core fetch agree automatically.
  2. Cap RAM to the backed DRAM. The RAM branch {4'd0,cpuaddr[27:0]} wraps 0x10000000–0x17FFFFFF back onto low DRAM; if DRAM backs less than the mapped range, a pattern-based sizer over-counts (see §8). Keep memcfg + addrmask consistent with what's actually backed.

Optional tighter layout (frees the wasted low 128 MB): relocate RAM to DRAM base (RAM 0x08000000+N → DRAM 0+N), put the device window at the top of the aperture. Not required at 128 MB.

7. Device plan: PL (real) vs ARM (emulated)

Timing-critical / tightly-polled devices become real memory-mapped peripherals in the PL; complex IRQ/DMA-driven peripherals are emulated on the ARM (latency-tolerant). The 0x1F… window stops being plain DRAM and becomes an AXI peripheral decode + a PS-bridge region.

Interrupts are not optional. The PL INT2 must aggregate sources (the 8254 tick, plus "done" signals the ARM raises for SCSI/net) and drive the core's extern_irq. Without it: you reach clock calibration but get no jiffies/scheduler tick and no I/O completion.

8. Memory sizing & the bus-error decision

Decision: do NOT implement a guest bus-error (DBE/IBE) path. Confirmed from two IRIX-capable references:
  • MAME sgi/mc.cpp (boots IRIX): no CPU bus error anywhere; memcfg_w dynamically installs RAM regions from whatever the firmware writes to the memory-config register. The OS learns memory size from memcfg, not from faults.
  • Real SGI PROM (IP32 firmware disasm, F_0x8100d2e0): the universal presence/size idiom is write 0xdeadbeef; read back; beql match — it branches on readback mismatch, never on an exception. Exactly the LFSR/pattern write-readback scheme.

What replaces the bus-error path (small, real requirements so a pattern probe of a hole reliably "misses"):

  1. RD_BAD returns 0 (r_load_data) — a defined value, not stale bus data. For the SGI PROM's 0xdeadbeef write/readback idiom this is a valid sentinel (0 ≠ pattern → readback misses → correctly "absent").
  2. No unmapped address may alias/wrap into real RAM — the other false-match route (the RAM-branch wrap, §6).
  3. memcfg must match the DRAM actually backed — it is the value the OS trusts.
Sentinel caveat: returning 0 is generically the least-robust choice — a zero-fill/march sizer that writes 0x00000000 would false-match a hole. The SGI PROM uses 0xdeadbeef/address signatures (never plain 0), so 0 is safe for the target firmware. If a future probe ever bites, a non-zero per-address sentinel (e.g. ~address) is bulletproof.

No guest bus-error path, no 4 GB mapping — just a compaction onto one RAM bank + a 16 MB device window with inert (zero-returning) holes.
