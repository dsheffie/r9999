## "synth"/"build"/"rebuild"/"deploy" is AMBIGUOUS — ASK which FPGA target: r9999 (core) vs henry (SoC)
There are **two different FPGA bitstreams**, and the user is inconsistent about naming. When the user says "synth"/"build"/"rebuild"/"deploy"/"the bitstream" WITHOUT disambiguating, **STOP and ask which one.** Aliases that DO disambiguate (don't ask): "core"/"mipscore"/"r9999" = r9999; "soc"/"henry" = henry.
- **r9999 / "core" / "mipscore":** `~/code/r9999/rebuild.sh` (→ `convert_sv_to_v.py`, `rebuild.tcl`) → project `~/fpga/ultra96v2-mipscore/`. A **bare core** with NO IP22 devices (no SCC/RTC/HPC3/MC); console = CP0 reg7 FIFO. Used for big-csmith / hello / cheri / randgen co-sim. **CANNOT boot IRIX/Linux** via `henry_arcs.bin` — the FSBL polls SCC RR0 @0xbfbd9830 and spins forever (no SCC).
- **henry / "soc":** `~/code/henry-the-wannabe-ip22-soc/gen_mipscore.sh` (FFs the `henry/r9999` **submodule** → `origin/main`, then sv2v of henry `rtl/*.sv` + r9999 core + `ip_hdl/` AXI/SCC-Rx wrapper) → `vivado -mode batch -source rebuild_henry.tcl -tclargs impl_11` → project `~/fpga/ultra96v2-henry/`. The **full IP22 SoC** (SCC console, ds1386 RTC, HPC3, MC). **Required for IRIX/Linux boot.** Deploy via `~/fpga/ultra96v2-henry/doit.sh`.
- **TRAP:** both projects output the SAME filename `ultra96v2_oob_wrapper.bit`, so deploying one **clobbers the other** on the board (the prior is backed up to `ultra96v2_oob_wrapper.prev.bit`). To put an r9999 change on silicon for IRIX you must build **henry** (it pulls the r9999 core via the submodule). Lesson learned 2026-06: a whole bisect was wasted building r9999/ultra96 to chase an "IRIX regression" that was just the no-SCC FSBL spin; `prev.bit` was a henry bit.

## dsheffie writes idiomatic Verilog — rely on it, and grep it for the easy check
Prefix convention: `r_*` = flop output (driven in an `always_ff` by `r_X <= n_X`). In an `always_comb` block, **`n_*` variables are the next state that updates the matching `r_*` flop**, and **`t_*` variables are combinational signals that do NOT drive a flop** (`w_*` = wire/comb). Block discipline follows: `always_ff @(posedge clk)` uses ONLY non-blocking `<=` and assigns ONLY `r_*`; `always_comb` (and `assign`) use ONLY blocking `=` and produce `n_*`/`t_*`. Because this is consistent, the bug check is a one-line grep: **an `always_ff` that uses `=` or assigns an `n_*`/`t_*`, or an `always_comb` that uses `<=`, is a bug.** That is exactly how the 64-bit mode hazard hid for so long — the Status next-state `n_sr_*` sat in an `always_ff` (blocking `=`) instead of `always_comb`. Verilator scheduled the race favorably so every sim passed; Vivado made the KX write ~50/50 metastable on silicon.

## Verilator passes but the FPGA fails (flaky / run-to-run) ⇒ sim/synth mismatch, NOT a logic bug

## OPEN: faulted-delay-slot serialize funnel — WAIT_FOR_SERIALIZE_IN_FAULTED_DELAY_SLOT faulted case (line ~1539) is currently UNREACHABLE (2026-06; for later consideration)
How we ENTER the funnel (proven by directed test, NOT timer-IRQ as an old memory note claimed): a **mispredicted branch** (`faulted=1` = the restart flag, `is_irq=0/is_ii=0`) whose **delay slot is a serializing op still in the decode queue**. Only `MTC0`/`DMTC0` set `serializing_op` (decode_mips.sv). A xorshift-low-bit branch (~50% unpredictable) with an `mtc0` delay slot hammers it: ~2000 entries/4000 iters, **co-sim checker clean** — the funnel's normal path (`WAIT -> ACTIVE`) is validated. randgen never hit it on its own only because its delay slots are always safe ALU (`ser=0`); real compiled code (kernel/csmith) puts `mtc0` in delay slots, which is the historical trigger.
The **faulted sub-case** (`WAIT` sees `t_rob_next_head.faulted` → `ARCH_FAULT`, the `//todo - is this correct?` branch) needs a serializing op that *itself* faults. That can't happen: the only serializing ops are `mtc0`/`dmtc0`, and their only fault path is a **decode-time RI** that rewrites `op -> II`, which **strips `serializing_op`** → it faults as an ordinary arch-exception OUTSIDE the funnel. decode-faults/IRQ all replace the op (non-serializing); `mtc0`/`dmtc0` take no execution-time fault (no mem access, **no CpU modeled for user-mode CP0 access**). So a "serializing AND faulted" op is structurally impossible here. **Decision needed:** either (a) accept line 1539 is defensive/dead and replace it with `assert(!t_rob_next_head.faulted)` to document the invariant (do NOT strip the funnel ENTRY/clean path — stripping it deadlocked randgen historically), or (b) if we ever want to exercise it, add an execution/commit-time RI that keeps the op serializing, or model CpU.
Related: added a decode-time 64b gate to `dmtc0` (now RIs in 32-bit mode like `daddu`; was mode-exempt). It does NOT enable the faulted case (RI→II→non-serializing). **Caveat:** carries Linux-boot regression risk if the kernel ever does `dmtc0` in 32-bit (KX=0) mode — verify a boot before trusting it. The randgen generator emits only `mtc0`/`mfc0` (never `dmtc0`), so the gate is not exercised by the co-sim sweep.
